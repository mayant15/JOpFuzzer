# JOpFuzzer Replication & Extension Plan

## Project Understanding

JOpFuzzer is a JIT compiler fuzzer that explores a **two-dimensional input space**:
1. **Java test programs** (generated by JavaFuzzer, mutated by TBar/JavaMutator)
2. **JIT compiler options** (78 HotSpot optimization flags ‚Äî booleans & numerics)

### Architecture Overview

```
MainEntry
  ‚îú‚îÄ‚îÄ Prepare.preprocess()         # Learn correlation matrix (EXPENSIVE)
  ‚îÇ     ‚îú‚îÄ‚îÄ Encode                 # Extract code features from regression tests
  ‚îÇ     ‚îú‚îÄ‚îÄ RelationMapping        # Run each test with each option on/off
  ‚îÇ     ‚îú‚îÄ‚îÄ Task                   # Execute profile diffs (LED algorithm)
  ‚îÇ     ‚îî‚îÄ‚îÄ saves result.csv       # SourceCodeFeature √ó Option matrix
  ‚îú‚îÄ‚îÄ SeedPool                     # Generate seeds via JavaFuzzer (Ruby)
  ‚îî‚îÄ‚îÄ MutationEntry                # Mutation loop
        ‚îú‚îÄ‚îÄ analyzeFile()          # Weighted statement selection for mutation
        ‚îú‚îÄ‚îÄ JavaMutator.jar        # TBar-based mutation (external jar)
        ‚îî‚îÄ‚îÄ DifferentialTest       # Test mutant under option combos
              ‚îú‚îÄ‚îÄ readArray()      # Load result.csv
              ‚îú‚îÄ‚îÄ analyzeChangedStructure()  # Select 3 options via matrix weights
              ‚îú‚îÄ‚îÄ executeOption()  # Enumerate 2^n configs, compare outputs
              ‚îî‚îÄ‚îÄ executeProfile() # Feedback: keep mutant if profiles diverge
```

### The Learned Model (What We Want to Replace)

The correlation matrix `result.csv` maps **33 code features** ‚Üí **78 JIT options**.

**How it's built** (`Prepare.preprocess()` + `RelationMapping.mapping()`):
1. For each regression test, extract methods and encode their features
2. For each method √ó option: run the test with profiling ON vs with the method excluded from JIT
3. Calculate List Edit Distance (LED) between the two profiles
4. Accumulate into `StructureOptionRelation[feature][option]`
5. Average across all samples

**How it's used** (`DifferentialTest.analyzeChangedStructure()`):
1. Extract features at the mutation point
2. For each option: sum correlation values for all changed features
3. Select 3 options via weighted random sampling (probability ‚àù weight)
4. Enumerate all 2^n value combinations for differential testing

**Problem**: The preprocessing step is **extremely slow** ‚Äî it runs `O(seeds √ó methods √ó 78 options √ó 2 executions)` JVM invocations. For the full compiler regression suite, this takes many hours even with parallelism.

### Coverage Measurement Notes

**What the paper measures (Section V-B):**
- Scope is `src/hotspot/` ‚Äî all HotSpot C/C++ source, excluding other JDK components
- The paper reports denominators of ~620 KLoc (JDK 11) and ~656 KLoc (JDK 17),
  which appear to be raw C++ code lines (cloc reports 620K `.cpp` lines for our JDK 17 source)
- lcov's "total lines" count is lower (~360K) because it only counts gcov-instrumented
  executable lines (excludes declarations, blank lines, comments, headers without executable code)
- The paper's percentages therefore use **raw LOC** as denominator, not lcov's executable line count
- Paper results for JOpFuzzer: **56.2% line / 47.6% function** (JDK 11), **57.5% line / 47.9% function** (JDK 17)
- Sub-components broken down: c1, c2 (opto), runtime, gc

**Our test run results (3 seeds, 5 mutations ‚Äî ~30 seconds):**

| Component | Lines Covered | Lines Total (lcov) | Line % | Func % |
|-----------|--------------|-------------------|--------|--------|
| All HotSpot | 169,553 | 360,262 | 47.1% | 43.4% |
| c1 | 14,180 | 18,984 | 74.7% | 78.7% |
| c2 (opto) | 44,217 | 67,725 | 65.3% | 70.4% |
| runtime | 11,099 | 23,431 | 47.4% | 56.5% |
| gc | 20,817 | 58,438 | 35.6% | 36.5% |

Note: Much of the baseline coverage comes from JVM startup, class loading, and JIT compilation
of the test programs. A longer fuzzing run (24h as in the paper) would incrementally increase
coverage in optimization-specific paths exercised by the mutated compiler options.

### Replicating the Paper's Full Experiment

The paper runs JOpFuzzer for **24 hours**, repeated **3 times**, measuring line and function
coverage of HotSpot C++ source via gcov/lcov. The script `Scripts/run-experiment.sh` automates this.

**Full replication (with learning step ‚Äî matches paper exactly):**
```bash
./Scripts/run-experiment.sh \
  --jdk-path ~/code/jdk17u-jdk-17.0.19-9/build/linux-x86_64-server-fastdebug/images/jdk \
  --jdk-source ~/code/jdk17u-jdk-17.0.19-9 \
  --regression-tests ~/code/jdk17u-jdk-17.0.19-9/test/hotspot/jtreg/compiler \
  --seed-count 100 \
  --duration-hours 24 \
  --repetitions 3 \
  --output-dir ./experiment-results
```

The learning step (`Prepare.preprocess()`) walks ~1674 `.java` files under the compiler
regression test directory, extracts code features, and runs each test with each of the 78
options on/off to build the correlation matrix `result.csv`. This is the most expensive part.

## Heuristic Replacement Strategy

### Goal
Replace the learned correlation matrix with **hard-coded heuristics** that map code features
to relevant JIT options, eliminating the expensive preprocessing step entirely.

### Design Rationale
The paper's key insight is that **specific optimizations are only relevant to code with matching
structures** (e.g., loop unrolling options only matter for code with loops). The learned matrix
essentially discovers these relationships empirically. But many of these relationships are
well-known from compiler engineering:

### Proposed Heuristic Rules

The mapping from `SourceCodeFeature` ‚Üí `Option` groups:

| Code Feature | Relevant Options | Rationale |
|---|---|---|
| `loopDepthOne/Two/ThreeOrMore` | Loop opts: `LoopMaxUnroll`, `LoopUnrollLimit`, `LoopUnrollMin`, `LoopOptsCount`, `LoopStripMiningIter`, `LoopPercentProfileLimit`, `MaxLoopPad`, `NumberOfLoopInstrToAlign`, `UseLoopPredicate`, `LoopUnswitching`, `PartialPeelLoop`, `PartialPeelNewPhiDelta`, `BlockLayoutRotateLoops`, `OptimizeFill` | Loop optimizations only apply to loops |
| `loopDepthOne+` + `arrayDimensionOne+` | Vectorization: `UseSuperWord`, `SuperWordLoopUnrollAnalysis`, `AlignVector` | Auto-vectorization targets array ops in loops |
| `fieldAccess` | Inlining: `Inline`, `InlineAccessors`, `InlineSmallCode`, `FreqInlineSize`, `ClipInlining`, etc. + EA: `DoEscapeAnalysis`, `EliminateAllocations` | Field access triggers accessor inlining and escape analysis |
| `synchronizedAccess` | Lock elimination: `EliminateLocks`, `EliminateNestedLocks` + `DoEscapeAnalysis` | Lock elision requires EA |
| `arithmeticOperator`, `shiftOperator` | Peephole: `OptoPeephole`, `CanonicalizeNodes` | Arithmetic simplification |
| `compareOperator` | `OptimizePtrCompare`, `DoCEE`, `EliminateNullChecks` | Comparison-related opts |
| `invocationStmt` | Inlining options (C1/C2 inline limits), `TieredCompilation` | Method calls ‚Üí inlining decisions |
| `tryStmt`, `throwStmt` | `InlineMethodsWithExceptionHandlers`, `EliminateBlocks` | Exception handling affects control flow |
| `arrayDimensionOne+` | `InlineArrayCopy`, `InlineObjectCopy`, `ArrayCopyLoadStoreMaxElem`, `ArrayOperationPartialInlineSize`, `MultiArrayExpandLimit`, `RangeCheckElimination` | Array operations |
| `constructorCallStmt` | `EliminateAllocations`, `EliminateAllocationArraySizeLimit`, `EliminateAllocationFieldsLimit`, `DoEscapeAnalysis`, `AggressiveUnboxing`, `EliminateAutoBox` | Object allocation opts |
| `lambda` | `IncrementalInline`, `IncrementalInlineMH`, `IncrementalInlineVirtual` | Lambda desugaring ‚Üí method handle inlining |
| `booleanType` + autoboxing | `EliminateAutoBox`, `AutoBoxCacheMax`, `AggressiveUnboxing` | Autoboxing related |
| **All features (baseline)** | `TieredCompilation`, `SplitIfBlocks`, `UseGlobalValueNumbering`, `UseLocalValueNumbering`, `DominatorSearchLimit`, `OptimizeUnsafes` | General optimization flags |

### Implementation Plan

1. **Create `HeuristicOptionSelector.java`** ‚Äî a new class that replaces
   `DifferentialTest.analyzeChangedStructure()` + `readArray()` with a hard-coded mapping
2. **Bypass `Prepare.preprocess()`** ‚Äî skip the preprocessing entirely when using heuristics
3. **Keep the same interface**: input = list of changed `SourceCodeFeature` names,
   output = list of 3 selected `Option` names
4. **Preserve weighted random selection**: some options should be more likely than others
   within each category (e.g., `LoopMaxUnroll` more impactful than `MaxLoopPad`)
5. **Optionally add category mixing**: always include at least one "general" option alongside
   feature-specific ones to increase diversity

### Changes Required

- `MainEntry.java`: Add flag to skip preprocessing, go straight to fuzzing
- `DifferentialTest.java`: Use `HeuristicOptionSelector` instead of `readArray()` + CSV
- New: `Options/HeuristicOptionSelector.java`
- Remove dependency on `result.csv` existing

## Observations & Future Work

### Observations
- The source layout was non-standard (flat `src/` instead of `src/main/java/`)
- The CLI arg names differ between README (`-seedPath`) and code (`-regressionTestPath`)
- `Mutation.DT.Task` has an unused import of `commons-compress` ‚Äî suggests the jar was
  bundled from a different project
- The `FreqInlineSize` option is referenced in the README but not in `Option.java` enum ‚Äî
  might be missing
- `DifferentialTest.executeProfile()` has a likely bug at line 98: it calls
  `Double.parseDouble(optionSimilarityList.get(i).getKey())` but the key is an option name
  string, not a number
- The `threshold` calculation in `readArray()` is suspicious ‚Äî it divides by `index * length`
  inside the loop, so the final threshold is the average of only the last row, not the global avg
- `executeOption()` enumerates `2^n` configurations which grows exponentially ‚Äî with 6 options
  (3 new + 3 origin) that's 64 JVM invocations per mutant
